## Context（背景）

gstack 的 design skills（/office-hours、/design-consultation、/plan-design-review、/design-review）目前都会产出 design 的 **文字描述**：带 hex codes 的 DESIGN.md、用 prose 写 pixel specs 的 plan docs、ASCII art wireframes。Creator 曾经在 OmniGraffle 中手工设计 HelloSign，他觉得这很丢脸。

Value unit 错了。用户不需要更丰富的 design language；他们需要一个可执行、可观看的 visual artifact，把 conversation 从“你喜欢这个 spec 吗？”变成“这就是 screen 吗？”。

## 问题陈述

Design skills 在描述 design，而不是展示 design。Argus UX overhaul plan 就是例子：487 行详细 emotional arc specs、typography choices、animation timing，却没有任何 visual artifacts。一个会“design”的 AI coding agent 应该产出你能看见、能立刻产生直觉反应的东西。

## 需求证据

Creator/primary user 觉得当前 output 很尴尬。每次 design skill session 都以 prose 收尾，而那里本该有 mockup。GPT Image API 现在可以生成 pixel-perfect UI mockups，且 text rendering 准确；过去用来 justify text-only output 的 capability gap 已经不存在。

## Narrowest Wedge（最窄切入点）

一个 compiled TypeScript binary（`design/dist/design`），wrap OpenAI Images/Responses API，并可通过 skill templates 中的 `$D` 调用（镜像现有 `$B` browse binary pattern）。优先集成顺序：/office-hours → /plan-design-review → /design-consultation → /design-review。

## Agreed Premises（已同意前提）

1. GPT Image API（via OpenAI Responses API）是正确 engine。Google Stitch SDK 是 backup。
2. **Visual mockups 对 design skills 默认开启**，并提供容易跳过的路径，而不是 opt-in。（按 Codex challenge 修订。）
3. Integration 是 shared utility（不是每个 skill 重写一遍），也就是一个任意 skill 都可调用的 `design` binary。
4. Priority：先 /office-hours，再 /plan-design-review、/design-consultation、/design-review。

## Cross-Model Perspective（Codex）

Codex 独立验证了 core thesis：“失败不在 markdown 内的 output quality，而是当前 value unit 错了。”关键贡献：
- Challenge premise #2（opt-in → default-on），accepted（已接受）
- 提出 vision-based quality gate：用 GPT-4o vision 检查 generated mockups 是否有 unreadable text、missing sections、broken layout，并自动 retry once
- 将 48-hour prototype 收窄为 shared `visual_mockup.ts` utility，仅覆盖 /office-hours + /plan-design-review，产出 hero mockup + 2 variants

## Recommended Approach（推荐方案）：`design` Binary（Approach B）

### 架构

**共享 browse binary 的 compilation and distribution pattern**（bun build --compile、setup script、skill templates 中的 $VARIABLE resolution），但 architecture 更简单：没有 persistent daemon server、没有 Chromium、没有 health checks、没有 token auth。Design binary 是 stateless CLI，负责调用 OpenAI API 并把 PNG 写到 disk。Multi-turn iteration 的 session state 放在 JSON file 中。

**New dependency（新增依赖）：** `openai` npm package（加入 `devDependencies`，不要放 runtime deps）。Design binary 与 browse 分开 compile，避免 openai bloat browse binary。

```
design/
├── src/
│   ├── cli.ts            # Entry point, command dispatch
│   ├── commands.ts        # Command registry (source of truth for docs + validation)
│   ├── generate.ts        # Generate mockups from structured brief
│   ├── iterate.ts         # Multi-turn iteration on existing mockups
│   ├── variants.ts        # Generate N design variants from brief
│   ├── check.ts           # Vision-based quality gate (GPT-4o)
│   ├── brief.ts           # Structured brief type + assembly helpers
│   └── session.ts         # Session state (response IDs for multi-turn)
├── dist/
│   ├── design             # Compiled binary
│   └── .version           # Git hash
└── test/
    └── design.test.ts     # Integration tests
```

### Commands（命令）

```bash
# 从 structured brief 生成 hero mockup
$D generate --brief "用于 coding assessment tool 的 dashboard。Dark theme，cream accents。展示：builder name、score badge、narrative letter、score cards。Target：technical users。" --output /tmp/mockup-hero.png

# 生成 3 个 design variants
$D variants --brief "..." --count 3 --output-dir /tmp/mockups/

# 基于 feedback 迭代 existing mockup
$D iterate --session /tmp/design-session.json --feedback "把 score cards 做大，把 narrative 移到 scores 上方" --output /tmp/mockup-v2.png

# 基于 vision 的 quality check（返回 PASS/FAIL + issues）
$D check --image /tmp/mockup-hero.png --brief "包含 builder name、score badge、narrative 的 dashboard"

# 带 quality gate + auto-retry 的 one-shot
$D generate --brief "..." --output /tmp/mockup.png --check --retry 1

# 通过 JSON file 传入 structured brief
$D generate --brief-file /tmp/brief.json --output /tmp/mockup.png

# 为 user review 生成 comparison board HTML
$D compare --images /tmp/mockups/variant-*.png --output /tmp/design-board.html

# Guided API key setup + smoke test（引导式 API key 设置与冒烟测试）
$D setup
```

**Brief input modes（brief 输入模式）：**
- `--brief "plain text"` — free-form text prompt（simple mode）
- `--brief-file path.json` — 匹配 `DesignBrief` interface 的 structured JSON（rich mode）
- Skills 构造 JSON brief file，写入 /tmp，然后传 `--brief-file`

**所有 commands 都注册在 `commands.ts` 中**，包括 `generate` 上的 `--check` 和 `--retry` flags。

### Design Exploration Workflow（设计探索工作流，from eng review）

Workflow 是 sequential，不是 parallel。PNGs 用于 visual exploration（human-facing），HTML wireframes 用于 implementation（agent-facing）：

```
1. $D variants --brief "..." --count 3 --output-dir /tmp/mockups/
   → Generates 2-5 PNG mockup variations

2. $D compare --images /tmp/mockups/*.png --output /tmp/design-board.html
   → Generates HTML comparison board (spec below)

3. $B goto file:///tmp/design-board.html
   → User reviews all variants in headed Chrome

4. User picks favorite, rates, comments, clicks [Submit]
   Agent polls: $B eval document.getElementById('status').textContent
   Agent reads: $B eval document.getElementById('feedback-result').textContent
   → 不需要 clipboard，也不需要 pasting。Agent 直接从页面读取 feedback。

5. Claude generates HTML wireframe via DESIGN_SKETCH matching approved direction
   → Agent 基于 inspectable HTML 实现，而不是基于 opaque PNG。
```

**Classifier: APP UI**（task-focused、utility page）。不做 product branding。

**Visual spec（视觉规格）：**
- Background: #fff。No shadows，no card borders。Variant separation：1px #e5e5e5 line。
- Typography：system font stack。Header：16px semibold。Labels：14px semibold。Feedback placeholder：13px regular #999。
- Star rating：5 clickable stars，filled=#000，unfilled=#ddd。不着色，不 animated。
- Radio button "Pick"：explicit favorite selection。每个 variant 一个，mutually exclusive。
- "More like this" button：per-variant，用该 variant style 作为 seed 触发 regeneration。
- Submit button：#000 background，white text，right-aligned。Single CTA。
- Regenerate bar：#f7f7f7 background，与 feedback area visually distinct。
- Mockup images max-width 1200px centered。Margins：24px sides。

**Interaction states（交互状态）：**
- Loading（page opens before images ready）：每个 card skeleton pulse with "Generating variant A..."。Stars/textarea/pick disabled。
- Partial failure（3 个中 2 个成功）：显示 good ones，失败项显示 error card 和 per-variant [Retry]。
- Post-submit："Feedback submitted! Return to your coding agent." Page stays open。
- Regeneration：smooth transition，fade out old variants、skeleton pulses、fade in new。Scroll resets to top。Previous feedback cleared。

**Feedback JSON structure**（written to hidden #feedback-result element）：
```json
{
  "preferred": "A",
  "ratings": { "A": 4, "B": 3, "C": 2 },
  "comments": {
    "A": "喜欢 spacing，header 感觉对了",
    "B": "太忙乱，但 color palette 不错",
    "C": "mood 完全不对"
  },
  "overall": "选 A，把 CTA 做大",
  "regenerated": false
}
```

**Accessibility（可访问性）：** Star ratings keyboard navigable（arrow keys）。Textareas labeled（"Feedback for Variant A"）。Submit/Regenerate keyboard accessible with visible focus ring。All text #333+ on white。

**Responsive（响应式）：** >1200px：comfortable margins。768-1200px：tighter margins。<768px：full-width，无 horizontal scroll。

**Screenshot consent（first-time only for $D evolve）：** "这会把你的 live site screenshot 发送给 OpenAI，用于 design evolution。[Proceed] [Don't ask again]" 存储在 ~/.gstack/config.yaml 的 design_screenshot_consent。

为什么 sequential：Codex adversarial review 指出 raster PNGs 对 agents 是 opaque（无 DOM、无 states、无 diffable structure）。HTML wireframes 保留回到 code 的 bridge。PNG 是给 human 说“是的，就是这个方向”的。HTML 是给 agent 说“我知道该怎么实现”的。

### Key Design Decisions（关键设计决策）

**1. Stateless CLI，不是 daemon**
Browse 需要 persistent Chromium instance。Design 只是 API calls，没有 server 的理由。Multi-turn iteration 的 session state 是写到 `/tmp/design-session-{id}.json` 的 JSON file，包含 `previous_response_id`。
- **Session ID:** 从 `${PID}-${timestamp}` 生成，通过 `--session` flag 传递
- **Discovery:** `generate` command 创建 session file 并打印 path；`iterate` 通过 `--session` 读取
- **Cleanup:** /tmp 中的 session files 是 ephemeral（OS 会清理）；无需 explicit cleanup

**2. Structured brief input**
Brief 是 skill prose 和 image generation 之间的 interface。Skills 从 design context 构造它：
```typescript
interface DesignBrief {
  goal: string;           // "Dashboard for coding assessment tool"
  audience: string;       // "Technical users, YC partners"
  style: string;          // "Dark theme, cream accents, minimal"
  elements: string[];     // ["builder name", "score badge", "narrative letter"]
  constraints?: string;   // "Max width 1024px, mobile-first"
  reference?: string;     // Path to existing screenshot or DESIGN.md excerpt
  screenType: string;     // "desktop-dashboard" | "mobile-app" | "landing-page" | etc.
}
```

**3. Design skills 默认开启**
Skills 默认 generate mockups。Template 包含 skip language：
```
Generating visual mockup of the proposed design... (say "skip" if you don't need visuals)
```

**4. Vision quality gate（视觉质量门）**
生成后，可选地把 image 送入 GPT-4o vision 检查：
- Text readability（labels/headings 是否 legible？）
- Layout completeness（requested elements 是否都存在？）
- Visual coherence（它看起来像 real UI，还是 collage？）
Failure 时 auto-retry once。如果仍 fail，带 warning 展示。

**5. Output location：explorations 在 /tmp，approved finals 在 `docs/designs/`**
- Exploration variants 去 `/tmp/gstack-mockups-{session}/`（ephemeral，不 commit）
- 只有 **user-approved final** mockup 保存到 `docs/designs/`（checked in）
- Default output directory 可通过 CLAUDE.md `design_output_dir` setting 配置
- Filename pattern：`{skill}-{description}-{timestamp}.png`
- 如果 `docs/designs/` 不存在则创建（mkdir -p）
- Design doc 引用 committed image path
- 始终通过 Read tool 展示给用户（Claude Code 中会 inline render images）
- 避免 repo bloat：只 commit approved designs，不 commit 每个 exploration variant
- Fallback：如果不在 git repo，保存到 `/tmp/gstack-mockup-{timestamp}.png`

**6. Trust boundary acknowledgment（信任边界确认）**
Default-on generation 会把 design brief text 发送到 OpenAI。这是相对现有 HTML wireframe path（entirely local）的 new external data flow。Brief 只包含 abstract design descriptions（goal、style、elements），不会包含 source code 或 user data。来自 $B 的 screenshots 不会发送到 OpenAI（DesignBrief 中的 reference field 是 agent 使用的 local file path，不上传到 API）。需要在 CLAUDE.md 中记录。

**7. Rate limit mitigation（速率限制缓解）**
Variant generation 使用 staggered parallel：通过带 delays 的 `Promise.allSettled()`，每个 API call 间隔 1 秒启动。这样避免 image generation 的 5-7 RPM rate limit，同时比 fully serial 快。如果任何 call 429，使用 exponential backoff retry（2s、4s、8s）。

### API Details（API 细节）

**Generate（生成）：** OpenAI Responses API with `image_generation` tool
```typescript
const response = await openai.responses.create({
  model: "gpt-4o",
  input: briefToPrompt(brief),
  tools: [{ type: "image_generation", size: "1536x1024", quality: "high" }],
});
// Extract image from response output items
const imageItem = response.output.find(item => item.type === "image_generation_call");
const base64Data = imageItem.result; // base64-encoded PNG
fs.writeFileSync(outputPath, Buffer.from(base64Data, "base64"));
```

**Iterate（迭代）：** Same API with `previous_response_id`
```typescript
const response = await openai.responses.create({
  model: "gpt-4o",
  input: feedback,
  previous_response_id: session.lastResponseId,
  tools: [{ type: "image_generation" }],
});
```
**NOTE（说明）：** 通过 `previous_response_id` 做 multi-turn image iteration 是需要 prototype validation 的 assumption。Responses API 支持 conversation threading，但是否会保留 generated images 的 visual context 用于 edit-style iteration，docs 中未确认。**Fallback（回退）：** 如果 multi-turn 不工作，`iterate` fallback 到用 original brief + accumulated feedback 在 single prompt 中重新 generate。

**Check（检查）：** GPT-4o vision
```typescript
const check = await openai.chat.completions.create({
  model: "gpt-4o",
  messages: [{
    role: "user",
    content: [
      { type: "image_url", image_url: { url: `data:image/png;base64,${imageData}` } },
      { type: "text", text: `Check this UI mockup. Brief: ${brief}. Is text readable? Are all elements present? Does it look like a real UI? Return PASS or FAIL with issues.` }
    ]
  }]
});
```

**Cost（成本）：** 每个 design session 约 $0.10-$0.40（1 hero + 2 variants + 1 quality check + 1 iteration）。相比每次 skill invocation 已有 LLM costs，可以忽略。

### Auth（认证，validated via smoke test）

**Codex OAuth tokens 不能用于 image generation。** 2026-03-26 测试：Images API 和 Responses API 都 reject `~/.codex/auth.json` access_token，错误为 "Missing scopes: api.model.images.request"。Codex CLI 也没有 native imagegen capability。

**Auth resolution order（认证解析顺序）：**
1. Read `~/.gstack/openai.json` → `{ "api_key": "sk-..." }`（file permissions 0600）
2. Fall back to `OPENAI_API_KEY` environment variable
3. 如果两者都不存在 → guided setup flow：
   - 告诉用户："Design mockups 需要拥有 image generation permissions 的 OpenAI API key。可在 platform.openai.com/api-keys 获取。"
   - Prompt 用户 paste key
   - 以 0600 permissions 写入 `~/.gstack/openai.json`
   - Run smoke test（generate a 1024x1024 test image）验证 key 有效
   - 如果 smoke test passes，继续。如果 fail，show error 并 fallback to DESIGN_SKETCH。
4. 如果 auth 存在但 API call fails → fallback to DESIGN_SKETCH（现有 HTML wireframe approach）。Design mockups 是 progressive enhancement，never hard requirement。

**New command（新命令）：** `$D setup` — guided API key setup + smoke test。可随时运行以更新 key。

## Assumptions to Validate in Prototype（prototype 中要验证的假设）

1. **Image quality:** "Pixel-perfect UI mockups" 是 aspirational。GPT Image generation 未必能稳定产出真正 UI fidelity 下准确 text rendering、alignment 和 spacing。Vision quality gate 有帮助，但在 full skill integration 前，需要 prototype validation “good enough to implement from” 这个 success criterion。
2. **Multi-turn iteration:** `previous_response_id` 是否保留 visual context 未证明（见 API Details section）。
3. **Cost model:** 估算 $0.10-$0.40/session 需要 real-world validation。

**Prototype validation plan（prototype 验证计划）：** Build Commit 1（core generate + check），跨不同 screen types 运行 10 个 design briefs，在进入 skill integration 前评估 output quality。

## CEO Expansion Scope（CEO 扩展范围，accepted via /plan-ceo-review SCOPE EXPANSION）

### 1. Design Memory + Exploration Width Control
- 从 approved mockups 自动 extract visual language 到 DESIGN.md
- 如果 DESIGN.md 存在，future mockups constrain 到 established design language
- 如果没有 DESIGN.md（bootstrap），跨 diverse directions WIDE explore
- Progressive constraint：design 越 established，exploration band 越窄
- Comparison board 获得带 exploration controls 的 REGENERATE section：
  - "Something totally different"（wide exploration）
  - "More like option ___"（围绕 favorite narrow）
  - "Match my existing design"（constrain to DESIGN.md）
  - 用于 specific direction changes 的 free text input
  - Regenerate refreshes the page，agent polls for new submission

### 2. Mockup diff
- `$D diff --before old.png --after new.png` 生成 visual diff
- Side-by-side，并 highlight changed regions
- 使用 GPT-4o vision 识别 differences
- Used in：/design-review、iteration feedback、PR review

### 3. Screenshot 到 mockup 演进

- 从 reality 开始，而不是 blank canvas

- 组合 $B screenshot + $D diff + vision analysis

### 5. Responsive variants
- `$D variants --brief "..." --viewports desktop,tablet,mobile`
- 自动在多个 viewport sizes 生成 mockups
- Comparison board 显示 responsive grid，支持 simultaneous approval
- 让 responsive design 从 mockup stage 开始成为 first-class concern

- 通过 vision analysis 从 approved PNG extract colors、typography、layout
- 与 DESIGN.md 和 HTML wireframe 组合成 structured spec
- Bridge "approved design" 到 "agent starts coding"，zero interpretation gap
