# Design: GStack Self-Learning Infrastructure（自学习基础设施）

Mode（模式）：Open Source / Community

## 问题陈述

GStack 跨 sessions 运行 30+ skills，但 sessions 之间不会学习。一次 /review session 抓到 N+1 query pattern，下一次在同一个 codebase 上运行 /review 又从零开始。一次 /ship run 发现 test command，以后每次 /ship 都重新发现。一次 /investigate 找到棘手 race condition，未来没有任何 session 知道它。

每个 AI coding tool 都有这个问题。Cursor 有 per-user memory。Claude Code 有 CLAUDE.md。Windsurf 有 persistent context。但它们都不会 compound。它们不结构化自己学到的东西。它们也不会跨 skills 共享知识。

## 我们在构建什么

一种 per-project institutional knowledge，能跨 sessions 和 skills compound。Structured、typed、confidence-scored learnings，每个 gstack skill 都能读写。目标：同一个 codebase 跑过 20 次 sessions 后，gstack 知道每个 architectural decision、每个 past bug pattern，以及每一次它曾经错在哪里。

## North Star（北极星目标）

/autoship（Release 5）。一个命令里装下一支完整 engineering team。描述 feature、approve plan，其他全部自动。/autoship 如果没有 learnings（R1）、review quality（R2）、session persistence（R3）和 adaptive ceremony（R4）就无法工作。Releases 1-4 是让 /autoship 真正可行的 infrastructure。

## Audience（受众）

使用 AI 构建产品的 YC founders。也就是每周在真实 codebases 上运行 gstack 20+ 次，并且会注意到它重复问同一个问题的人。

## Differentiation（差异化）

| Tool（工具） | Memory model（memory 模型） | Scope（范围） | Structure（结构） |
|------|-------------|-------|-----------|
| Cursor | Per-user chat memory | Per-session | Unstructured |
| CLAUDE.md | Static file | Per-project | Manual |
| Windsurf | Persistent context | Per-session | Unstructured |
| **GStack** | **Per-project JSONL** | **Cross-session, cross-skill** | **Typed, scored, decaying** |

---

## State Systems（状态系统）

gstack 有四个不同的 persistence layers。它们共享 storage patterns（`~/.gstack/projects/$SLUG/` 中的 JSONL），但服务不同目的：

| System（系统） | File（文件） | What it stores（存储内容） | Written by（写入者） | Read by（读取者） |
|--------|------|---------------|------------|---------|
| **Learnings** | `learnings.jsonl` | Institutional knowledge（pitfalls、patterns、preferences） | All skills | All skills（preamble） |
| **Timeline** | `timeline.jsonl` | Event history（skill start/complete、branch、outcome） | Preamble（automatic） | /retro、preamble context recovery |
| **Checkpoints** | `checkpoints/*.md` | Working state snapshots（decisions、remaining work、files） | /checkpoint、/ship、/investigate | Preamble context recovery、/checkpoint resume |
| **Health** | `health-history.jsonl` | Code quality scores over time（per-tool、composite） | /health | /retro、/ship（gate）、/health（trends） |

它们不重叠。Learnings = 你知道什么。Timeline = 发生过什么。Checkpoints = 你在哪里。Health = code 有多好。每个 system 回答不同问题。

## Release Roadmap（发布路线图）

### Release 1: "GStack Learns"（GStack 学习）(v0.13-0.14) — SHIPPED

**Headline（标题）：** 每个 session 都让下一个 session 更聪明。

已 shipped：
- Learnings persistence 位于 `~/.gstack/projects/{slug}/learnings.jsonl`
- 用于 manual review、search、prune、export 的 `/learn` skill
- 所有 review findings 上的 confidence calibration（1-10 scores with display rules）
- Observed/inferred learnings 的 confidence decay（1pt/30d）
- Cross-project learnings discovery（opt-in, AskUserQuestion consent）
- Reviews 匹配 past learnings 时显示 "Learning applied" callouts
- 集成进 /review、/ship、/plan-*、/office-hours、/investigate、/retro

Schema（schema）：
```json
{
  "ts": "2026-03-28T12:00:00Z",
  "skill": "review",
  "type": "pitfall",
  "key": "n-plus-one-activerecord",
  "insight": "Always check includes() for has_many in list endpoints",
  "confidence": 8,
  "source": "observed",
  "branch": "feature-x",
  "commit": "abc1234",
  "files": ["app/models/user.rb"]
}
```

Types（类型）：`pattern` | `pitfall` | `preference` | `architecture` | `tool`
Sources（来源）：`observed` | `user-stated` | `inferred` | `cross-model`

架构：append-only JSONL。Duplicates 在 read time resolve（每个 key+type 的 "latest winner"）。没有 write-time mutation，没有 race conditions。

### Release 2: "Review Army"（Review Army）(v0.14.3-0.14.4) — SHIPPED

**Headline（标题）：** 每个 PR 都有 10 个 specialist reviewers。

已 shipped：
- 7 个 parallel specialist subagents：always-on（testing、maintainability）+
  conditional（security、performance、data-migration、API contract、design）+
  red team（large diffs / critical findings）
- JSON-structured findings，带 confidence scores + across agents 的 fingerprint dedup
- 每次 review 记录 PR quality score（0-10）+ /retro trending
- Learning-informed specialist prompts，按 domain 注入 past pitfalls（过去踩过的坑）
- Multi-specialist consensus highlighting，多位 specialist 确认的 findings 会被 boosted
- 通过 PLAN_COMPLETION_AUDIT 增强 Delivery Integrity
- Checklist refactored：CRITICAL categories 留在 main pass，specialist categories 抽取到 review/specialists/ 中的 focused checklists

### Release 2.5: "Review Army Expansions"（Review Army 扩展）— NOT YET SHIPPED

**Headline（标题）：** R2 证明稳定后再 ship。检查 core loop 表现如何。

Pre-check（前置检查）：review R2 quality metrics（PR quality scores、specialist hit rates、false positive rates、E2E test stability）。如果 core loop 有问题，先修它们。

要 ship 的内容：
- E1：Adaptive specialist gating，auto-skip 0-finding track record 的 specialists。通过 gstack-learnings-log 存储 per-project hit rates。用户可用 --security 等强制运行。
- E3：Test stub generation，每个 specialist 在 findings 旁输出 TEST_STUB。从 project 检测 framework（Jest/Vitest/RSpec/pytest/Go test）。流入 Fix-First：AUTO-FIX applies fix + creates test file。
- E5：Cross-review finding dedup，读取 gstack-review-read 获取 prior review entries。Suppress 匹配 prior user-skipped finding 的 findings。
- E7：Specialist performance tracking，通过 gstack-review-log 记录 per-specialist metrics。Timeline integration：specialist runs 出现在 timeline.jsonl 中，用于 /retro trending。

### Release 3: "Session Intelligence"（Session Intelligence）(v0.15.0) — SHIPPED

**Headline（标题）：** 你的 AI sessions 会记住发生过什么。

已 shipped：
- Session timeline：每个 skill 自动把 start/complete events 记录到 `~/.gstack/projects/$SLUG/timeline.jsonl`。Local-only，never sent anywhere，不受 telemetry setting 影响，always on。
- Context recovery：compaction 后或 session start 时，preamble 列出 recent CEO plans、checkpoints 和 reviews。Agent 读取最新文件来 recover context。
- Cross-session injection：preamble 为 current branch 打印 LAST_SESSION 和 LATEST_CHECKPOINT。你在输入任何内容前就能看到上次停在哪里。
- Predictive skill suggestion：如果你最近 3 次 sessions 遵循某个 pattern（review、ship、review），gstack 会建议你下一步可能想做什么。
- Session start 时合成 "Welcome back" context message。
- `/checkpoint` skill：save/resume/list working state snapshots。支持跨 branch listing，用于 Conductor workspace handoff between agents。
- `/health` skill：code quality scorekeeper，包裹 project tools（tsc、biome、knip、shellcheck、tests）。Composite 0-10 score、trend tracking、scores drop 时给 improvement suggestions。
- Timeline binaries：`bin/gstack-timeline-log` 和 `bin/gstack-timeline-read`。
- Routing rules：/checkpoint 和 /health 添加到 preamble skill routing。

Design doc（设计文档）：`docs/designs/SESSION_INTELLIGENCE.md`

### Release 4: "Adaptive Ceremony"（Adaptive Ceremony）— NOT YET SHIPPED

**Headline（标题）：** GStack 尊重你的时间，同时不牺牲 safety。

Ceremony 和 trust 是不同 concerns。Ceremony = 一个 PR 会经历的 review/test/QA steps 集合。Trust = 决定适用哪个 ceremony level 的 policy engine。它们相互作用，但不合并。

要 ship 的内容：

**Ceremony levels（ceremony 层级）：**
- FULL：all specialists、adversarial、Codex structured review、coverage audit、plan completion。用于 large diffs、new features、migrations、auth changes。
- STANDARD：adversarial + Codex、coverage audit、plan completion。用于 medium diffs、typical feature work。
- FAST：adversarial only。用于 trusted projects 上 small、well-tested changes。

**Trust policy engine（trust policy engine）：**
- Scope-aware trust。Trust 按 change class earned，不是 globally earned。Docs-only PRs 上的 clean history 不会为 migration PRs 买到 trust。
- Change class detection：docs、tests、config、frontend、backend、migrations、auth、infra。每个 class 有自己的 trust threshold。
- Trust signals：consecutive clean reviews（per class）、/health score stability、regression frequency、test coverage trends。
- Trust never fast-tracks：migrations、auth/permission changes、new API endpoints、infrastructure changes。这些无论 trust level 如何都始终获得 FULL ceremony。
- Gradual degradation，不是 binary reset。单个 regression 不会 reset all trust。它只会让该 change class 的 trust 降一级。

**Scope assessment（scope 评估）：**
- /review、/ship、/autoplan 中基于 diff size、files touched 和 change class 做 TINY/SMALL/MEDIUM/LARGE classification。
- Ceremony level = f(scope, trust, change class)。

**TODO lifecycle（TODO 生命周期）：**
- /triage 用于 incoming TODOs 的 interactive approval
- /resolve 通过 parallel agents 做 batch resolution

### Release 5: "/autoship — One Command, Full Feature"（一个命令，完整 feature）— NOT YET SHIPPED

**Headline（标题）：** 描述一个 feature。Approve plan。其他全部自动。

/autoship 是 resumable state machine，不是 linear pipeline。Review 和 QA 可以把 work 送回 build/fix。Compaction 可以 interrupt 任意 phase。系统必须能 graceful recover。

```
                    ┌──────────┐
                    │  START   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │ /office- │
                    │  hours   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │/autoplan │ ◄── single approval gate
                    └────┬─────┘
                         │
              ┌──────────▼──────────┐
              │       BUILD         │ ◄── /checkpoint auto-save
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │      /health        │ ◄── quality gate
              │   (score >= 7.0)    │
              └──────────┬──────────┘
                         │ fail → back to BUILD
              ┌──────────▼──────────┐
              │      /review        │
              └──────────┬──────────┘
                         │ ASK items → back to BUILD
              ┌──────────▼──────────┐
              │        /qa          │
              └──────────┬──────────┘
                         │ bugs found → back to BUILD
              ┌──────────▼──────────┐
              │       /ship         │
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │ /checkpoint archive │ ◄── preserve, don't destroy
              └─────────────────────┘
```

要 ship 的内容：
- /autoship autonomous pipeline，使用上面的 state machine。
  每个 phase 写入 timeline.jsonl。每个 phase 前 checkpoints auto-save。
  Compaction recovery：context recovery 读取 checkpoint + timeline，从 last completed phase resume。
- Completion 时 archive checkpoint（不是 deletion）。Recovery state 会保留，用于 debugging failed autoship runs。
- /ideate brainstorming skill（parallel divergent agents + adversarial filtering）
- /plan-eng-review 中的 research agents（codebase analyst、history analyst、best practices researcher、learnings researcher）

Depends on：R1（research agents 所需 learnings）、R2（quality 所需 review army）、R3（persistence 所需 session intelligence）、R4（speed 所需 adaptive ceremony）。

### Release 6: "Execution Studio"（执行工作室）— NOT YET SHIPPED

**Headline（标题）：** Parallel execution infrastructure。

要 ship 的内容：
- Swarm orchestration：multi-worktree parallel builds。基于 /checkpoint（R3）的 Conductor workspace handoff。一个 orchestrator skill 把 independent workstreams dispatch 给 parallel agents，每个 agent 有自己的 worktree。
- Codex build delegation：按 task type（boilerplate、test generation、mechanical refactors）auto-detect 何时 delegate implementation 给 Codex CLI。
- PR feedback resolution：跨 review platforms 的 parallel comment resolver。
- /onboard：从 codebase analysis 自动生成 contributor guide。
- /triage-prs：maintainers 的 batch PR triage。

### Release 7: "Design & Media"（Design 与 Media）— NOT YET SHIPPED

**Headline（标题）：** Visual design integration。

要 ship 的内容：

- Cross-platform portability（Copilot、Kiro、Windsurf output）

## 风险登记

### Proxy signals as permission to skip scrutiny（代理信号被误用为跳过审查的许可）
（Identified by Codex review, 2026-04-01）

/health scores、clean review history 和 timeline patterns 是有用 signals。它们不是 safety 的 proof。如果这些 signals 同时 feed ceremony reduction 和 /autoship，failure mode 是 rare、silent、high-severity mistakes。Mitigations：
- 某些 change classes 永远不 fast-track（migrations、auth、infra、new endpoints）。
- Trust degrades gradually，不是 binary reset。
- /autoship 在每个 project 第一次运行时始终执行 FULL ceremony。Trust 是 earned 的。

### Stale context recovery（过期 context recovery）
（Identified by Codex review, 2026-04-01）

Context recovery 可能注入 wrong-branch state、obsolete plans 或 invalid checkpoints。Mitigations：
- Checkpoints 在 YAML frontmatter 中包含 branch name。Context recovery 按 current branch filter。
- Timeline grep 在显示 LAST_SESSION 前按 branch filter。
- Stale artifact detection：如果 checkpoint 超过 7 天，标记为 potentially stale，而不是当作 current 呈现。

### Validation metrics needed（需要验证指标）
（Identified by Codex review, 2026-04-01）

Shipping R4（Adaptive Ceremony）前，需要 measure：
- Predictive suggestion accuracy（用户是否运行了 suggested skill？）
- Trust policy false-skip rate（fast-tracked PRs 是否有 post-merge issues？）
- Context recovery accuracy（recovered context 是否匹配 actual state？）
- /health score correlation with actual code quality（高 scores 是否预测 fewer production bugs？）

这些 metrics 应在 R3 usage 期间收集，并在 R4 ship 前 review。
